<task type="auto">
  <name>Task 1: Add failure_category to apply_result and error_result with per-path wiring</name>
  <files>src/dispatch.rs</files>
  <action>
  1. Add a `failure_category: Option<&str>` parameter to both `apply_result` and `error_result` functions. Include it in the JSON output as `"failure_category": failure_category`.

  2. For success paths (status="ok"), pass `None` for failure_category. The JSON output will have `"failure_category": null`.

  3. Map each failure exit point in run_apply_flow to a category. Based on source inspection:

     | Exit point | Event logged | Category |
     |---|---|---|
     | `model_status_error` | model returned error status | `"model"` |
     | `missing_code` | response missing code field | `"schema"` |
     | `preflight_unresolved_imports` (exhausted) | unresolved crate imports | `"preflight"` |
     | `verify_command_error` | verify command failed to run | Check if error is timeout (from tokio::time::timeout) -> `"timeout"`, otherwise -> `"verify"` |
     | `verify_failed` (exhausted all attempts) | verify checks failed | `"verify"` |
     | `apply_without_verify` (success) | applied without verify | pass `None` |
     | `verify_passed` (success) | applied and verified | pass `None` |

  4. For the non-apply path (normalize_non_apply_output), add failure_category to the output JSON. If status is "ok", set null. If status is "error", set "unknown" as fallback (the non-apply path does not have structured failure categories yet).

  5. For network/HTTP errors from send_request in dispatch_with_retry, the error propagates up. If the function catches and converts these to error_result, add `"network"` category. If they propagate as Err(), the caller may need to catch and wrap. Check the actual error flow.

  6. For format retry exhaustion in dispatch_with_retry (FORMAT_RETRIES exceeded), if this converts to an error_result, add `"format"` category.

  7. Update ALL call sites of apply_result and error_result to pass the appropriate failure_category.

  8. Add unit tests:
     - test_apply_result_includes_failure_category: verify JSON output includes failure_category field
     - test_error_result_includes_failure_category: verify JSON output includes failure_category field
     - test_apply_result_success_null_category: success path has null failure_category

  9. Run cargo fmt, cargo clippy --workspace --all-targets -- -D warnings, cargo test. Fix any issues.

  NOTE: apply_result already has #[allow(clippy::too_many_arguments)]. Adding one more parameter stays within the existing suppression. If the suppression bothers the reviewer, consider refactoring to a struct parameter, but keep the change focused per handoff guidance.
  </action>
  <verify>
  - Every apply_result call site passes a failure_category
  - Every error_result call site passes a failure_category
  - Success paths pass None; failure paths pass the correct category string
  - cargo test passes with all new and existing tests
  - cargo clippy passes clean (no new warnings)
  - No new #[allow] attributes added beyond the existing too_many_arguments
  - JSON output includes "failure_category" key in all dispatch results
  </verify>
