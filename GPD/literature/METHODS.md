## Scope and Evidentiary Boundary

This methods survey answers a narrow question: what methods are appropriate for determining whether Awl's bounded local Ollama dispatch workflow saves paid frontier-model tokens in real Claude/Codex coding sessions without unacceptable pass-rate or reliability regressions.

It does not establish that savings exist. The inspected Awl artifacts show a functional local-dispatch harness and a partial three-task 7B run, but no same-task frontier baseline. Therefore the current evidence supports method selection and gap identification only.

## Recommended Methods

### Analytical Methods

| Method | Purpose | Why Recommended for Awl |
|---|---|---|
| Paired same-task A/B comparison | Compare direct-frontier and Awl-assisted outcomes on identical tasks | Controls task difficulty variance better than comparing aggregate sessions with different work. |
| Intent-to-treat accounting | Count every task assigned to an arm, including failures and abandoned recoveries | Prevents hiding failed local dispatches by excluding them after the fact. |
| Per-task cost decomposition | Split frontier packaging, local worker usage, compact-result review, and failure recovery | Awl can only claim net savings after orchestration overhead and recovery costs are included. |
| Pass-rate guardrail analysis | Require savings and correctness thresholds simultaneously | A low-cost workflow that silently shifts semantic debugging back to the frontier is not a success. |
| Failure taxonomy | Classify failures as `format`, `schema`, `preflight`, `verify`, `timeout`, `network`, or `unknown` | Converts failures into actionable decisions: tool fix, model opt-up, frontier takeover, or benchmark exclusion. |
| Model-selection calibration | Compare 7B-only and 14B-only results before writing frontier routing guidance | Avoids treating 14B as automatically better or 7B as automatically economical without evidence. |

### Empirical A/B Methods

| Method | Purpose | Implementation in This Project |
|---|---|---|
| Same-task direct frontier baseline | Measure paid tokens when Claude/Codex solves each task without Awl | Create `experiments/results/baseline.csv` with one row per task and split input/output token fields. |
| Awl-assisted arm | Measure frontier overhead plus local dispatch result for the same task | Use `experiments/run_awl_arm.sh` after retry, model override, and telemetry patches land. |
| 7B-only and 14B-only sweeps | Estimate pass-rate/cost/latency tradeoff by model | Run the same task pack twice with exact model strings recorded. Do not mix model policies inside a sweep. |
| Stratified task pack | Avoid a benchmark that only favors 7B-easy work | Include Python and Rust; write-from-scratch, edit-existing, and context-paths-required tasks; easy/medium/hard tags. |
| Paired statistical summaries | Quantify uncertainty on savings and pass-rate differences | Use paired bootstrap confidence intervals for token savings and McNemar or exact paired tests for pass/fail differences when sample size grows. |
| Human/session replay log | Capture actual frontier turns needed for packaging, review, acceptance, and recovery | Needed because Awl's product claim concerns real Claude/Codex sessions, not isolated local model calls. |

Minimum next empirical design:

- Start with at least 10 mixed tasks, as already recommended by the project reports.
- Prefer 20 or more before making durable guidance about model selection.
- Preserve the existing `01_string_helper` trailing-newline failure; it is a useful known edge case, not a task to tune away.
- Run every task in direct-frontier, 7B-only Awl, and 14B-only Awl modes if budget permits.
- Record whether the frontier accepted the Awl result as-is, needed log inspection, re-dispatched, or took over manually.

### Telemetry and Cost Accounting

| Quantity | Required Field | Method |
|---|---|---|
| Frontier baseline input tokens | `frontier_input_tokens` | Capture from Claude/Codex billing or session telemetry for the direct run. |
| Frontier baseline output tokens | `frontier_output_tokens` | Capture separately; do not blend with input tokens. |
| Awl orchestration input tokens | `awl_frontier_input_tokens` | Count task packaging prompt, context selection, compact-result review, and any failure recovery prompt. |
| Awl orchestration output tokens | `awl_frontier_output_tokens` | Count frontier responses around dispatch, review, and recovery. |
| Local worker prompt tokens | `prompt_tokens` or `input_tokens` | Already present in `experiments/run_awl_arm.sh` and dispatch logs. |
| Local worker completion tokens | `completion_tokens` or `output_tokens` | Already present in `experiments/run_awl_arm.sh` and dispatch logs. |
| Local worker wall time | `wall_ms`, `model_ms` | Retain for latency tradeoff analysis even if local tokens are not paid frontier tokens. |
| Verification outcome | `checks_passed`, `attempts`, `open_issues` | Interpret success through the verifier, not only model status. |
| Failure category | `failure_category` | Add to `apply_result` and `error_result`; aggregate in `scripts/dispatch_cost_report.py`. |

Cost equation for each task:

```text
direct_frontier_cost =
  frontier_input_tokens * input_price_per_token
  + frontier_output_tokens * output_price_per_token

awl_frontier_cost =
  awl_frontier_input_tokens * input_price_per_token
  + awl_frontier_output_tokens * output_price_per_token

frontier_cost_saved = direct_frontier_cost - awl_frontier_cost
```

Local Ollama tokens should be reported, but not converted to paid frontier cost unless the study explicitly assigns a local compute cost model. The main Awl thesis is paid frontier-token savings, so local tokens are a secondary load and latency metric.

Project-specific accounting requirements:

- Replace blended `--cost-per-mtok` reporting in `experiments/tally.py` with `--input-cost-per-mtok` and `--output-cost-per-mtok`.
- Do the same for `scripts/dispatch_cost_report.py`.
- Treat the contract's `$5 input / $25 output per MTok` as the current agreed analysis assumption, but re-check provider price cards before publishing final cost claims.
- Report savings both in tokens and dollars; token savings alone can mislead when output tokens are priced differently.
- Include failed Awl attempts in aggregate cost. Excluding them would bias the result toward savings.

### Software Verification and CI Methods

| Method | Purpose | Awl-Specific Application |
|---|---|---|
| Apply-verify-rollback invariant tests | Ensure failed local work does not dirty the workspace | Keep and extend dispatch snapshot/write/verify/restore tests. |
| Mocked dispatch integration tests | Exercise stdin JSON to model response to write/verify/result schema without requiring Ollama | Add deterministic fixtures for success, verify failure, malformed schema, timeout, and network error. |
| Shell/path safety tests | Protect bounded delegation from escaping the workspace or running unsafe commands | Add focused `src/safety.rs` tests for allowlist, traversal, pipelines, redirects, and forbidden fragments. |
| Response sanitization tests | Validate code-fence stripping and JSON recovery behavior | Add `src/llm_io.rs` tests for malformed JSON, fenced JSON, escaped strings, and control characters. |
| Experiment script fixture tests | Protect the measurement tooling itself | Add fixtures for `experiments/tally.py` and `scripts/dispatch_cost_report.py`, including split I/O pricing and failure-category rollups. |
| Cross-platform CI | Ensure benchmark scaffolding and Rust gates are portable | Preserve Ubuntu/macOS CI with `cargo fmt --check`, clippy `-D warnings`, `cargo test`, and package dry run. |

Reliability patch planning should be staged in this order:

1. Change verified apply default retry from 2 to 1 and update tests.
2. Remove or justify task-level `max_attempts` overrides before rerunning Step 1.
3. Add per-dispatch `model` override through Rust dispatch options, CLI, MCP schema, and `AWL_MODEL_OVERRIDE`.
4. Add `failure_category` to dispatch outputs and reports.
5. Add split input/output cost accounting.
6. Only then resume the A/B sweep.

This order avoids measuring the old retry policy, blocked model-selection design, and obsolete cost convention.

### Benchmark Design Controls

| Control | Rationale | Concrete Rule |
|---|---|---|
| Pre-register task inclusion | Prevent cherry-picking tasks after model behavior is observed | Freeze task IDs, descriptions, setup, tests, language, and difficulty before running arms. |
| Same verifier across arms | Avoid giving one arm easier acceptance criteria | Direct frontier and Awl-assisted runs must use the same `verify_command`. |
| No 7B tuning | Avoid biasing the benchmark toward the local model | Do not simplify prompts/tests because 7B failed. Preserve meaningful edge cases. |
| Balanced task strata | Avoid a Python-only, pure-function-only conclusion | Include language, task type, context requirement, and difficulty labels. |
| Clean sandbox per run | Avoid contamination across arms | Use idempotent `setup.sh`; recreate sandboxes before every arm. |
| Exact model identity | Avoid informal model labels | Record full model strings such as `qwen2.5-coder:7b-instruct-q4_K_M` and `qwen2.5-coder:14b`. |
| Fixed retry policy | Avoid cost/pass-rate confounding | Use one verified apply attempt by default; any opt-in retry must be recorded as a separate configuration. |
| Blinded task order where feasible | Reduce session-order and operator learning effects | Randomize task order per arm or alternate arms by task. |
| Equal context budget policy | Avoid favoring one arm with richer context | Define what files/context the frontier may inspect directly and what Awl may receive in `context_paths`/repomap. |

Benchmark families to learn from, not copy blindly:

- HumanEval supports function-level pass-rate thinking but is too narrow for Awl's real product claim.
- EvalPlus shows why weak tests overstate correctness; Awl task tests should include edge cases and negative cases.
- SWE-bench motivates real repository edits, but full SWE-bench is likely too broad for the first Awl savings experiment.
- RealHumanEval motivates measuring human/frontier interaction overhead rather than relying only on static benchmark pass rates.

## Method Selection by Problem Type

**If the task is a bounded single-file edit with a strong local verifier:**

- Use Awl-assisted A/B measurement.
- This is the regime where Awl's apply/verify/rollback design can plausibly save frontier tokens.

**If the task is architecture-heavy, ambiguous, or weakly testable:**

- Use direct frontier as the default baseline and mark Awl dispatch as exploratory only.
- Awl may burn local time and still force frontier semantic review, so savings claims are fragile.

**If the task requires context from multiple files but has a focused acceptance test:**

- Use `context_paths` plus `auto_repomap`, and record whether context was sufficient.
- Treat failures as evidence about grounding and task packaging, not only model capability.

**If the task fails due to `verify`:**

- Do not same-model retry by default.
- Classify whether the failure looks like model capability, insufficient context, weak prompt, flaky test, or verifier defect.
- Let frontier decide between takeover and re-dispatch with 14B.

**If the task fails due to `format`, `schema`, or `preflight`:**

- Keep local reconciliation where cheap and high-yield.
- Track separately from semantic failures because these suggest tooling fixes rather than model opt-up.

## Validation Strategy by Method

| Method | Validation Approach | Key Benchmarks or Checks |
|---|---|---|
| A/B token savings | Paired per-task direct vs Awl comparison; aggregate and per-stratum reports | Same task IDs, same verifier, split I/O tokens, bootstrap intervals. |
| Pass-rate guardrail | Compare `checks_passed` and frontier accepted-as-is rate | Threshold from project reports: at least 60-70% usable Awl pass rate before positive guidance. |
| Failure taxonomy | Fixture tests plus manual audit of early failures | Every failure must have exactly one primary category and optional notes. |
| Model-selection guidance | Compare 7B-only and 14B-only sweeps by task stratum | Identify strata where 14B improves pass rate enough to justify latency. |
| Reliability patches | Unit, integration, and CI tests | `cargo fmt --check`, clippy `-D warnings`, `cargo test`, cost-report fixtures, optional Ollama smoke. |
| Cost scripts | Golden-file or JSON fixture tests | Verify split input/output pricing and failed-task inclusion. |

## Known Limitations and Failure Modes

| Limitation or Failure Mode | Why It Matters | Mitigation |
|---|---|---|
| No frontier baseline exists yet | The core savings claim is unmeasured | Collect `baseline.csv` or equivalent split-token baseline before any conclusion. |
| Current task pack is three Python tasks | Too small and narrow for guidance | Expand to at least 10 mixed tasks, preferably 20+ for model-selection claims. |
| Existing task JSON overrides retry counts | It can preserve the old expensive retry policy even after code default changes | Remove or isolate overrides before the next Step 1 run. |
| Verifier is the only oracle | Weak tests can make wrong code look successful | Strengthen tests; include edge cases; add mutation-style checks where cheap. |
| Frontier overhead is easy to undercount | Packaging/review/recovery may erase local savings | Record all frontier turns in the assisted arm. |
| Local compact summaries may be insufficient | Opening logs can add frontier tokens | Track when log inspection was required and include those tokens. |
| 14B performance is currently unmeasured | Model-selection guidance would be speculative | Run separate 14B-only sweep after override support lands. |
| Task selection can bias results toward Awl | Easy bounded tasks may not represent user workflows | Pre-register strata and report scope of validity explicitly. |
| Provider tokenizers and prices can change | Dollar estimates may drift | Store raw split token counts and rerun cost conversion with current rates. |
| Wall time is not the target metric | Faster or slower local runs do not directly imply paid-token savings | Report latency as secondary; do not use it as a substitute for token accounting. |

## Alternatives Considered

| Recommended | Alternative | When to Use Alternative |
|---|---|---|
| Paired task A/B | Aggregate before/after session comparison | Only for exploratory telemetry; too confounded for claims. |
| Static 7B-only and 14B-only sweeps | Dynamic auto-escalation | Defer dynamic escalation; user-approved design says frontier chooses upfront. |
| Strong local verifier | Human review only | Use human review for ambiguous tasks, but do not count it as local correctness proof. |
| Split input/output pricing | Blended cost per MTok | Blended rates are acceptable only for rough preflight estimates, not final reporting. |
| Pre-registered task pack | Iteratively tuning tasks after failures | Use tuning only when explicitly creating a new benchmark version, never mid-run. |

## What Not to Use

| Avoid | Why | Use Instead |
|---|---|---|
| Claiming savings from local worker tokens alone | The product claim is paid frontier-token reduction | Measure direct frontier versus Awl-assisted frontier token spend. |
| Excluding failed Awl tasks from cost summaries | It creates survivorship bias | Intent-to-treat aggregate over every attempted task. |
| Treating `status: ok` as sufficient | The relevant outcome is verified correctness and frontier acceptance | Use `checks_passed`, verifier logs, and accepted-as-is status. |
| Tuning tasks to rescue 7B pass rate | It biases the benchmark toward the local model | Preserve failures and classify them. |
| Comparing 7B and 14B on different task sets | It confounds model capability with task difficulty | Run the exact same frozen task pack. |
| Publishing dollar estimates without rate/date metadata | Token prices and tokenizers change | Store raw tokens, model names, price assumptions, and analysis date. |
