  <name>Task 1: Implement iterative quicksort</name>
  <files>sorting/quicksort.py</files>
  <action>
    Create sorting/quicksort.py implementing:

    ```
    def quicksort(arr: list) -> list:
    ```

    The implementation MUST be iterative (no recursion). Exact structure:

    1. Handle edge cases: if len(arr) <= 1, return list(arr).

    2. Create a working copy: `work = list(arr)` — input must not be mutated.

    3. Median-of-three pivot helper `_median3(arr, lo, hi)`:
       - mid = (lo + hi) // 2
       - Sort arr[lo], arr[mid], arr[hi] in-place among themselves (3-element sort, 3 conditional swaps)
       - Move the median to arr[hi] (so Lomuto can use it as pivot)
       - This eliminates worst-case O(n^2) behavior on sorted and reverse-sorted inputs

       Exact logic (ensure arr[hi] ends up as the median):
       ```
       mid = (lo + hi) // 2
       if arr[lo] > arr[mid]:
           arr[lo], arr[mid] = arr[mid], arr[lo]
       if arr[lo] > arr[hi]:
           arr[lo], arr[hi] = arr[hi], arr[lo]
       if arr[mid] > arr[hi]:
           arr[mid], arr[hi] = arr[hi], arr[mid]
       # Now arr[lo] <= arr[mid] <= arr[hi]; arr[hi] is max, arr[lo] is min
       # We want the median (arr[mid]) at arr[hi] for Lomuto
       arr[mid], arr[hi] = arr[hi], arr[mid]
       # Wait — that puts the max at hi. Re-examine:
       # After sorting: arr[lo] <= arr[mid] <= arr[hi]
       # arr[hi] is already the maximum; we want the MEDIAN at arr[hi] for Lomuto pivot
       # Correct: swap arr[mid] and arr[hi-1] (place median near end), use arr[hi-1] as pivot
       ```

       ALTERNATIVE (simpler, correct): place median at arr[hi-1]:
       ```python
       def _median3(arr, lo, hi):
           mid = (lo + hi) // 2
           # Sort the three candidates in-place
           if arr[lo] > arr[mid]:   arr[lo],  arr[mid] = arr[mid],  arr[lo]
           if arr[lo] > arr[hi]:    arr[lo],  arr[hi]  = arr[hi],   arr[lo]
           if arr[mid] > arr[hi]:   arr[mid], arr[hi]  = arr[hi],   arr[mid]
           # arr[lo] <= arr[mid] <= arr[hi]; place median adjacent to hi
           arr[mid], arr[hi-1] = arr[hi-1], arr[mid]
           return arr[hi-1]  # return the pivot value
       ```
       Then Lomuto partition runs on arr[lo+1 .. hi-1] with pivot = arr[hi-1].

       SIMPLEST CORRECT APPROACH (recommended — avoids fencepost confusion):
       ```python
       def _median3(arr, lo, hi):
           """Sort arr[lo], arr[mid], arr[hi] and place median at arr[hi]. Return pivot value."""
           mid = (lo + hi) // 2
           if arr[lo] > arr[mid]:   arr[lo],  arr[mid] = arr[mid],  arr[lo]
           if arr[lo] > arr[hi]:    arr[lo],  arr[hi]  = arr[hi],   arr[lo]
           if arr[mid] > arr[hi]:   arr[mid], arr[hi]  = arr[hi],   arr[mid]
           # Now arr[lo] <= arr[mid] <= arr[hi]. arr[hi] is the MAX.
           # We want the MEDIAN at arr[hi] for Lomuto. Swap arr[mid] and arr[hi].
           arr[mid], arr[hi] = arr[hi], arr[mid]
           # Now arr[hi] = original arr[mid] = median.
           # arr[lo] <= arr[hi] and arr[mid] = original arr[hi] (the max, now interior).
           # This is fine — Lomuto will sort everything in [lo..hi].
           return arr[hi]
       ```

       Note: the simplest correct approach puts the maximum at arr[hi], which is also valid for Lomuto — the pivot is arr[hi], and even if it's the maximum, Lomuto correctly places it at its final position. Median-of-three does NOT have to place the median at arr[hi]; it just needs to avoid picking a pathological pivot. Placing the max is safe but suboptimal. The standard approach is: after sorting the three, put the median at arr[hi] by swapping arr[mid] with arr[hi].

       FINAL RECOMMENDATION — this is the correct median-of-three that places the median at arr[hi]:
       ```python
       def _median3(arr, lo, hi):
           mid = (lo + hi) // 2
           # Ensure arr[lo] <= arr[mid]
           if arr[lo] > arr[mid]:
               arr[lo], arr[mid] = arr[mid], arr[lo]
           # Ensure arr[lo] <= arr[hi]
           if arr[lo] > arr[hi]:
               arr[lo], arr[hi] = arr[hi], arr[lo]
           # Now arr[lo] is the minimum of the three.
           # Ensure arr[mid] <= arr[hi]
           if arr[mid] > arr[hi]:
               arr[mid], arr[hi] = arr[hi], arr[mid]
           # Now arr[lo] <= arr[mid] <= arr[hi].
           # arr[mid] is the median. Place it at arr[hi] for Lomuto pivot.
           arr[mid], arr[hi] = arr[hi], arr[mid]
           # arr[hi] = median (the pivot). arr[mid] = old arr[hi] (the max, will be sorted correctly).
           return arr[hi]
       ```

    4. Lomuto partition `_partition(arr, lo, hi)`:
       - pivot = arr[hi] (placed there by _median3, or directly if subarray is small)
       - i = lo - 1
       - For j in range(lo, hi): if arr[j] <= pivot: i += 1; swap arr[i] and arr[j]
       - swap arr[i+1] and arr[hi]  (place pivot at its final position)
       - return i + 1  (pivot's final index)

       For subarrays of size <= 2 (hi - lo <= 1), skip median3 and partition directly:
       if hi - lo == 1 and arr[lo] > arr[hi]: swap them.

    5. Main iterative loop:
       ```python
       stack = [(0, len(work) - 1)]
       while stack:
           lo, hi = stack.pop()
           if hi - lo <= 0:
               continue
           if hi - lo == 1:
               if work[lo] > work[hi]:
                   work[lo], work[hi] = work[hi], work[lo]
               continue
           _median3(work, lo, hi)
           p = _partition(work, lo, hi)
           left_size  = p - 1 - lo
           right_size = hi - (p + 1)
           # Push larger partition first (processed later), smaller second (processed next)
           if left_size > right_size:
               stack.append((lo, p - 1))
               stack.append((p + 1, hi))
           else:
               stack.append((p + 1, hi))
               stack.append((lo, p - 1))
       return work
       ```

    CRITICAL: Do NOT use recursion. Do NOT call sys.setrecursionlimit. The only call stack depth is Python's own function call for quicksort() plus _median3() and _partition() — these are O(1) depth, not O(n).

    The push-smaller-first discipline MUST be implemented as shown: the larger partition goes onto the stack first (deeper in the stack), the smaller goes second (processed first on the next pop). This bounds the stack to O(log n) entries.

    Add a module docstring explaining the variant: "Iterative quicksort with explicit stack, median-of-three pivot selection, Lomuto partition scheme, and push-smaller-first stack discipline. O(n log n) average time, O(log n) stack space."
